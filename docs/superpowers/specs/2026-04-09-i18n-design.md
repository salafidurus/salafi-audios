# i18n Design Spec

## Overview

**Goal:** Add full internationalisation to the Salafi Durus platform — both UI string
translation and DB-level content translation — with Arabic and English as the initial
supported locales, full RTL layout support for Arabic, and a draft/publish approval
workflow for content translations.

**In scope:**

- UI strings (buttons, labels, navigation, empty states, errors) translated in both
  `apps/web` and `apps/mobile`.
- Content translations (lecture titles, scholar names, bios, series/collection titles,
  topic names) stored in the DB with per-translation approval status.
- RTL layout flip for Arabic across both apps.
- Locale detection for unauthenticated users from browser/device locale.
- Admin UI on both web and mobile for editors to create, save, and publish translations.

**Out of scope:**

- Languages beyond Arabic and English (system is designed to add more later).
- Machine translation or auto-translation integrations.
- Slug translation (slugs are routing keys and remain in the content's original language).
- Open user-submitted translations.

---

## Architecture

The system has four layers that build on each other:

```text
packages/core-i18n          ← locale constants, utilities, shared translation files
packages/core-contracts     ← updated DTOs (locale-resolved fields, translation DTOs)
apps/api                    ← locale interceptor, translation repo helpers, admin endpoints
apps/web + apps/mobile      ← i18next init, RTL layout, TranslationEditor admin component
```

## Section 1: Database

### Locale standard

Replace freeform language string fields with a `Locale` enum across the following
models:

- `Lecture.language: String?` → `Locale?`
- `Series.language: String?` → `Locale?`
- `Collection.language: String?` → `Locale?`
- `Scholar.mainLanguage: String?` → `Locale?` (note: the field is named `mainLanguage`
  on `Scholar`, not `language`)
- `LivestreamChannel.language: String?` → `Locale?`
- `User.preferredLanguage: String?` → `Locale?`

New enums:

```prisma
enum Locale {
  en
  ar
}

enum TranslationStatus {
  draft
  published
}
```

`TranslationStatus` is intentionally simpler than the existing `Status` enum —
translations have no review or archive lifecycle. A draft translation is invisible to
public consumers; a published one is live. No intermediate state is needed.

A data migration script normalises existing string values before the column type
change (e.g. `"Arabic"` → `ar`, `"English"` → `en`, unrecognised values → `null`).
The column type change itself is a breaking migration; deploy behind a maintenance
window or feature flag.

### Translation tables

Five new tables, one per translatable entity. All share the same structural pattern.

#### `ScholarTranslation`

```prisma
model ScholarTranslation {
  id        String            @id @default(cuid())
  scholarId String
  locale    Locale
  name      String
  bio       String?
  status    TranslationStatus @default(draft)
  createdAt DateTime          @default(now())
  updatedAt DateTime          @updatedAt

  scholar Scholar @relation(fields: [scholarId], references: [id], onDelete: Cascade)

  @@unique([scholarId, locale])
  @@index([scholarId, status])
}
```

### Fallback rule

The application layer (not the DB) resolves the final string:

1. Fetch only `published` translations. Draft translations are never passed to the
   resolution helper and are never visible to public or authenticated non-admin consumers.
2. If a published translation exists for the requested locale → use translated fields.
3. If none exists → use the entity's own primary fields (`title`, `name`, `bio`, etc.).

Content is always displayable — a missing translation is never a hard error.

## Section 2: API

### Locale resolution — `LocaleInterceptor`

A NestJS interceptor at `apps/api/src/shared/interceptors/locale.interceptor.ts` runs
on every request and attaches the resolved locale to `request.locale`. The
`apps/api/src/shared/` directory already exists; create `interceptors/` as a new
subdirectory within it.

Resolution order:

1. `?locale=` query param.
2. Authenticated user's `preferredLanguage` (from JWT/session).
3. `Accept-Language` request header, mapped to a supported locale via `resolveLocale()`
   from `@sd/core-i18n`.
4. Default: `en`.

### Translation resolution helper

A pure utility function at `apps/api/src/shared/utils/resolve-translated-fields.ts`,
used in every content repo:

```typescript
export function resolveTranslatedFields<T extends Record<string, string | null | undefined>>(
  primaryFields: T,
  publishedTranslation: Partial<T> | null,
): T {
  if (!publishedTranslation) return primaryFields;
  return Object.fromEntries(
    Object.entries(primaryFields).map(([key, value]) => [
      key,
      publishedTranslation[key as keyof T] ?? value,
    ]),
  ) as T;
}
```

Callers **must** pass only a `published` translation (or `null`) — never a draft.
This is enforced by the repo query which filters `status: 'published'` before passing
the result to this function.

### Admin translation endpoints — module placement

The API currently has no standalone `series` or `collections` modules. Translation
management endpoints for all five entity types are placed as follows:

- `Scholar` translations → `apps/api/src/modules/scholars/` (add new controller methods)
- `Lecture` translations → `apps/api/src/modules/lectures/` (add new controller methods)
- `Series` translations → `apps/api/src/modules/scholars/` (series are scholar-scoped;
  add series translation routes here as `/scholars/:scholarId/series/:seriesId/translations`)
- `Collection` translations → `apps/api/src/modules/scholars/` (same rationale;
  `/scholars/:scholarId/collections/:collectionId/translations`)
- `Topic` translations → `apps/api/src/modules/topics/` (add new controller methods)

All endpoints are protected by `manage:content` permission.

Using scholars as the canonical example:

```text
GET    /scholars/:id/translations
         → { translations: TranslationViewDto[] }

POST   /scholars/:id/translations
         body: { locale: Locale; name: string; bio?: string | null }
         → TranslationViewDto (status: draft)

PATCH  /scholars/:id/translations/:locale
         body: { name?: string; bio?: string | null }
         → TranslationViewDto

POST   /scholars/:id/translations/:locale/publish
         → TranslationViewDto (status: published)

POST   /scholars/:id/translations/:locale/unpublish
         → TranslationViewDto (status: draft)
```

No new top-level routes — translations are always sub-resources of their parent entity.

### User locale preference endpoint

There is currently no `account` module. Add a `PATCH /auth/me/locale` endpoint to the
existing `apps/api/src/modules/auth/` module:

```text
PATCH  /auth/me/locale
         body: { preferredLanguage: Locale }
         → { preferredLanguage: Locale }
         auth: required (any authenticated user)
```

This is a minimal addition to the auth module — no new module is created.

### No breaking contract changes

All existing DTOs return locale-resolved strings at the same field names (`title`,
`description`, `name`, `bio`). Consumers do not need to know a translation was applied.
The `language` / `mainLanguage` fields on content DTOs change type from
`string | undefined` to `Locale | undefined`.

## Section 3: `packages/core-i18n`

### Package exports map

The `package.json` for `@sd/core-i18n` must declare deep import paths so apps can
import locale JSON files directly:

```json
{
  "exports": {
    ".": "./src/index.ts",
    "./locales/*": "./dist/locales/*"
  }
}
```

The locale JSON files live in `src/locales/` but are published under `dist/locales/`,
because `package.json` sets `"files": ["dist"]`. The build script must copy them:
add a `cp -r src/locales dist/locales` step after the TypeScript compile step in
`packages/core-i18n/package.json`'s `"build"` script. This ensures
`import enLocale from '@sd/core-i18n/locales/en.json'` resolves correctly in webpack 5
(web) and Metro (mobile).

### Source structure

```text
packages/core-i18n/src/
  supported-locales.ts   ← SUPPORTED_LOCALES, DEFAULT_LOCALE, RTL_LOCALES, Locale type
  locale-utils.ts        ← isRtl(), resolveLocale(), localeToDir()
  index.ts               ← re-exports all of the above
  locales/
    en.json              ← shared UI strings, namespaced by feature
    ar.json              ← Arabic UI strings, same structure
```

### `supported-locales.ts`

```typescript
export const SUPPORTED_LOCALES = ["en", "ar"] as const;
export type Locale = (typeof SUPPORTED_LOCALES)[number];
export const DEFAULT_LOCALE: Locale = "en";
export const RTL_LOCALES: Locale[] = ["ar"];
```

### `locale-utils.ts`

```typescript
export function isRtl(locale: Locale): boolean;
export function localeToDir(locale: Locale): "ltr" | "rtl";
export function resolveLocale(candidate: string | null | undefined): Locale;
// resolveLocale maps any Accept-Language string or device locale tag to a
// supported locale, falling back to DEFAULT_LOCALE.
```

### Locale file structure

Both `en.json` and `ar.json` use the same top-level namespace keys:

```json
{
  "common": {},
  "auth": {},
  "feed": {},
  "lecture": {},
  "scholar": {},
  "search": {},
  "library": {},
  "live": {},
  "admin": {},
  "legal": {},
  "support": {}
}
```

### App-local overrides

Each app maintains a small override namespace for strings that differ by platform
(e.g. `"tap"` vs `"click"`):

```text
apps/web/src/core/i18n/overrides.en.json
apps/web/src/core/i18n/overrides.ar.json
apps/mobile/src/core/i18n/overrides.en.json
apps/mobile/src/core/i18n/overrides.ar.json
```

These are registered with higher priority than the shared locale files. Both apps import
the shared files from `@sd/core-i18n/locales/en.json` (and `ar.json`) via the declared
exports map.

### i18next initialisation

Each app has its own `i18n.ts` bootstrap:

```text
apps/web/src/core/i18n/i18n.ts
apps/mobile/src/core/i18n/i18n.ts
```

Both call `i18next.init` with:

- `resources`: shared locale files merged with local overrides.
- `fallbackLng: 'en'`.
- `lng`: resolved from `resolveLocale()` at startup.
- `interpolation.escapeValue: false` (React handles escaping).

## Section 4: RTL Layout

### Web (`apps/web`)

- `localeToDir(locale)` is called in `apps/web/src/app/layout.tsx`.
- Sets `<html lang={locale} dir={dir}>` on every render.
- All CSS (including `apps/web/src/app/globals.css` and every CSS Module) uses CSS
  logical properties exclusively:
  - `padding-inline-start` / `padding-inline-end` instead of `padding-left` / `padding-right`.
  - `margin-inline-start` / `margin-inline-end`.
  - `border-inline-start` / `border-inline-end`.
  - `inset-inline-start` / `inset-inline-end` for positioned elements.
  - `text-align: start` / `text-align: end` instead of `left` / `right`.
- No per-component RTL special-casing is needed for the majority of components.
- Icons that have directional meaning (arrows, chevrons, back buttons) use a CSS
  `transform: scaleX(-1)` applied conditionally via `[dir="rtl"] .icon-directional`.

### Mobile (`apps/mobile`)

- On locale change, call `I18nManager.forceRTL(isRtl(locale))` then
  `Updates.reloadAsync()` to restart the JS bundle.
- React Native automatically mirrors flex direction, text alignment, and touch handling
  when `I18nManager.isRTL` is `true`.
- All layout styles use `start`/`end` never `left`/`right` (enforced by a custom ESLint
  rule added to `packages/util-config/eslint/expo.js`).

## Section 5: Admin Translation UI

### `TranslationEditor` component

Implemented independently in each app (different UI primitives) but backed by the same
domain hooks. Lives at:

```text
apps/web/src/features/admin/components/TranslationEditor/
apps/mobile/src/features/admin/components/TranslationEditor/
```

**Behaviour:**

- Renders a locale tab strip (`EN` | `AR`).
- Displays the translatable fields for the entity type (e.g. `title` + `description`
  for a lecture; `name` + `bio` for a scholar).
- When a translation does not yet exist for a locale, pre-populates the fields
  read-only and visually dimmed with the original content as a translation reference.
- Status badge: `draft` (amber) or `published` (green).
- **Save draft** — available to any editor with `manage:content`.
- **Publish** / **Unpublish** — available to editors with `manage:content`.

**Placement:** A "Translations" tab inside the existing content detail/edit screen for
each entity. No separate top-level translation management screen.

### Domain hooks (in `packages/domain-content`)

These hooks call admin-protected endpoints and require an authenticated session. They
use the same auth context (`httpClient` from `@sd/core-contracts`) as existing domain
hooks in the package — no new auth dependency is introduced.

```typescript
useContentTranslations(entityType: TranslatableEntity, entityId: string)
  → { translations: TranslationViewDto[]; isLoading: boolean; error: unknown }

useSaveTranslation(entityType: TranslatableEntity, entityId: string)
  → { save: (locale: Locale, fields: Record<string, string | null>) => Promise<void>; isSaving: boolean }

usePublishTranslation(entityType: TranslatableEntity, entityId: string)
  → { publish: (locale: Locale) => Promise<void>; unpublish: (locale: Locale) => Promise<void> }
```

`TranslatableEntity = 'scholars' | 'lectures' | 'series' | 'collections' | 'topics'`

## Section 6: Locale preference — user flow

### Unauthenticated

- **Web:** At SSR time, read `Accept-Language` from the request header and resolve via
  `resolveLocale()`. Store the result in a `locale` cookie so subsequent requests are
  consistent without repeating the header parse. The `<html dir>` attribute is set from
  this cookie value.
- **Mobile:** At app boot, call `resolveLocale(getLocales()[0].languageCode)` from
  `expo-localization`. Store in `AsyncStorage` under key `'locale'`.

### Authenticated

- On login, if the user has no `preferredLanguage` set, write the currently active
  locale to `User.preferredLanguage` via `PATCH /auth/me/locale`.
- A locale picker (accessible from the account/settings screen on both apps) allows
  the user to override their preference at any time. Changing it calls
  `PATCH /auth/me/locale` with `{ preferredLanguage: locale }` and triggers an RTL
  reload on mobile if the direction changes (`isRtl(newLocale) !== isRtl(oldLocale)`).

## Testing strategy

- **`packages/core-i18n`:** `resolveLocale.spec.ts` covers common `Accept-Language`
  values (`en-US`, `ar`, `ar-SA`, `fr` → fallback `en`), `null`, and empty string.
- **API unit tests:** `locale.interceptor.spec.ts` covers all four resolution cases
  (query param, user preference, header, default). Each repo method that uses
  `resolveTranslatedFields` has a spec with three cases: published translation present
  (returns translated fields), draft translation present (falls back to primary — caller
  must not pass drafts), no translation (returns primary fields).
- **API integration tests:** each translation endpoint (`POST`, `PATCH`, `publish`,
  `unpublish`, `GET`) tested against a real DB in the module's
  `*.integration.spec.ts`.
- **Web/mobile component tests:** `TranslationEditor.spec.tsx` tests the draft
  pre-population (dimmed reference state), save flow (calls `useSaveTranslation`), and
  publish flow (calls `usePublishTranslation`), all with mocked hooks.
- **RTL smoke (Playwright on web):** Set the `locale` cookie to `ar` before navigating
  to `/`. Assert `document.documentElement.dir === 'rtl'` and that a known directional
  element (e.g. the sidebar) has `padding-inline-start` instead of `padding-left` in
  its computed style. Do not use the `?locale=` query param — that is an API-layer
  concern, not a Next.js locale concern.

## Migration path

1. Add `Locale` and `TranslationStatus` enums and the five translation tables
   (non-breaking — new tables only).
2. Run a data migration script to normalise existing `language` / `mainLanguage` string
   values to `Locale` enum values.
3. Change column types to use the `Locale` enum (breaking migration — deploy behind a
   maintenance window or feature flag).
4. Deploy API with `LocaleInterceptor` (defaults to `en` — no behaviour change for
   existing clients).
5. Ship UI i18n with English strings first (identical to current hardcoded strings —
   no visible change).
6. Ship Arabic UI strings + RTL layout.
7. Editors enter Arabic content translations via admin UI; publish when ready.
